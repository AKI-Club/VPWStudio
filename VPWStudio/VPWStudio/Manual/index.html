Introduction

VPW Studio is a tool that assists in ROM hacking the games in AKI Corporation's Virtual Pro-Wrestling series.

Supported Games

Currently, only the AKI wrestling games on the Nintendo 64 are supported:

Program Options

Tools → Program Options (todo)

Emulator

The "Emulator" section of the Program Options dialog allows you to set up an emulator to use for testing the changed ROM.

  • Path – Path to the emulator executable file.
  • Arguments – Any additional arguments to be passed to the emulator.

Build

The "Build" portion of the Program Options dialog only contains an option to modify the verbosity of the build log. The default option is "Normal".

Project Files

VPW Studio primarily operates on the concept of Project Files. The use of project files stems from the number of total conversion hacks created by the AKI hacking communities.

(more todo)

Creating a New Project

Getting started with the main portion of the program requires creating a new project. Select "New Project..." from the File menu, which will bring up the New Project dialog.

(todo)

The options presented are:

  • Project Name – The name of this project.
  • Author(s) – Names of people involved in the project. Optional.
  • Game Type – Set the game type.
  • Base ROM File – Set the location of the base Z64 ROM.
  • Output ROM File – Set the location of the built Z64 ROM.
  • Custom Location File – When checked, allows you to use a custom Location File. Helpful if you've already made changes to the base ROM that result in differences from the default Location File for the specified Game Type.

All of these may be changed later in the Project Settings dialog. Upon pressing OK, you will be sent back to the main screen.

Open an Existing Project

To open a VPW Studio Project File, choose "Open Project..." from the File menu. After selecting a project file, the program interface will update to reflect what game is being edited.

Save, Save As

When you want to save changes to a project, there are two options: "Save Project", and "Save Project As". Both are available from the File menu.

Save Project

The first time you save a new project, the program will ask you for a location and name to save the file. Upon doing so, two directories will be created in the directory where the project file is saved: "Assets" and "ProjectFiles". "Assets" is meant for unconverted items (such as .png images), while "ProjectFiles" will store any files that are in game-native formats.

Save Project As

"Save Project As" is similar to saving normally, but it always asks for a filename and doesn't automatically create the "Assets" and "ProjectFiles" directories.

Build ROM

To build a ROM with the changes you've made, select "Build ROM" from the "Project" menu. This will bring up a build log window, which provides updates on the build process.

Play ROM

If you want to test the changes in an emulator, select "Play ROM" from the "Project" menu. If the changed ROM hasn't been built yet, it will build the ROM before playing it.

For this to work, you must set up the emulator details in the Program Options section.

The File Table editor, accessed from "Project → File Table", allows you to view and edit the entries in the game's File Table. A large portion of the game's data is found in the File Table, including wrestler movesets, parameters, move data, textures, palettes, models, and more.

Appendix A: Glossary

The Virtual Pro-Wrestling series hacking communities use jargon to refer to some items and concepts. These terms can be confusing when first encountered. In addition, certain programs are involved whose name and function are not immediately obvious.

Addy
Short for "address", as in "memory address".
Broad Action
The general action the wrestler is performing.
ID2
One of the two ways of identifying a wrestler. This value is one byte and controls indexing into various arrays and structures. ID2 values are always given in hex.
ID4
One of the two ways of identifying a wrestler. This value is two bytes, being used as a unique identifier (not necessarily matching the ID2 order). ID4 values are always given in hex.
MidwayDec
A tool by Zoinkity that handles various N64 archive formats.
Offsetter
Another Zoinkity-created tool for inserting files into a VPW2 ROM and fixing the relevant offsets.
Toki 1
(todo; related to move effects)
Toki 2
(todo; related to moves)
Toki 3
Toki 3 values are related to items in the FileTable. Depending on who you ask, they're either the file IDs themselves, or the location of the data in the ROM (relative to the start location of the first file in the archive).

Appendix B1: LocationFile Format

The LocationFile format is a plain text database defining locations in ROM and RAM to be used by VPW Studio. Each non-comment line consists of a entry in the LocationFile.

Lines starting with the # character are comments, and will be ignored by the LocationFile loader.

Breakdown

LocationType:Location,Size;Comment

LocationType

This value is either ROM or RAM. ROM locations assume the Z64 file format (like the rest of the program). RAM locations are primarily for use with the GameShark code tool.

Location

The location for this entry, specified in hex. If this is a RAM entry, it must begin with 80. Do not use 81 if you are converting a GameShark code to a LocationFile entry; the size is handled in the next section.

Size

The size of this entry, in decimal. For a normal entry, this is how many bytes to read/write.

When handling a Code Change entry (see below), the Size value represents the number of bytes between the first and second change.

Comment

The comment section is typically used to provide a note about the location, but comments starting with special characters have different meanings.

Special characters currently handled:

  • $ – Data Location: Specific locations VPW Studio looks for to handle ROM reads/writes.
  • % – Code Change: Locations used for modifying game code.

Special location strings are mapped in the program, so you can't make things up and expect VPW Studio to understand them. The current list of handled special location strings:

Data Locations
  • $FILETABLE – Location of the FileTable in ROM.
  • $FIRSTFILE – Location of the first file described by the FileTable in ROM. (Size field does not matter.)
  • $WRESTLERDEFS – Location of Wrestler Definitions.

Appendix B2: FileTableDB Format

The FileTableDB format is a plain text database mapping File IDs to types and notes.

Lines starting with the # character are comments, and will be ignored by the FileTableDB loader.

Breakdown

FileID=Type;Notes

FileID

The ID file for this entry. Entries are in hex and should have 4 digits. Entries start from 0001, as 0000 is used as a "no file" index.

Type

(This is the challenging part.)

Currently supported values:

  • Binary – Default file type.
  • AkiArchive – An archive of files. Zoinkity's Midwaydec uses "TEXarchive" for this format, but not all archives exclusively contain TEX files.
  • AkiModel – Model data.
  • AkiTexture – AKI texture wrapper (a.k.a. "TEX")
  • AkiText – AKI text archive.
  • AkiLargeFont – Large font data (24px wide)
  • AkiSmallFont – Small font data (16px wide)
  • AkiFontChars – Font Characters (typically Shift-JIS)
  • Ci4Palette – CI4 palette (16 colors * 2 bytes)
  • Ci8Palette – CI8 palette (256 colors * 2 bytes)
  • I4Texture – I4 texture (4BPP)
  • Ci4Texture – CI4 texture (4BPP)
  • Ci8Texture – CI8 texture (8BPP)
  • DoubleTex – WCW/nWo Revenge credits faces (two TEX files, one after another)

Notes

A note displayed in the FileTable dialog. Unlike the LocationFile format, there are no special tokens to deal with.
